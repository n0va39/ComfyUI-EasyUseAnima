0.1.9에서는 0.1.7/0.1.8 이후 확인된 버그를 정리하고, Prompt Studio 계열에 wildcard 기능을 추가한다.

- 개발 중 이전 판단과 충돌하는 내용은
  [`docs/development/current-policies.md`](../development/current-policies.md)를

- Impact Pack `DetailerForEach`의 detailer core는 재구현하지 않는다.
- 저장된 결과물 workflow는 wildcard가 확장된 동일 결과를 재현할 수 있어야 한다.

- 사용자가 기존에 관리하던 wildcard 폴더도 EasyUse Anima 설정에서 추가 경로로 등록해 함께 사용할 수 있게 한다.
- sequential wildcard는 랜덤 선택이 아니라 seed modulo index 기반 순차 선택으로 구현한다.
- Prompt Studio Advanced UI에서는 기존 `mod guidance` 제어 아래에 wildcard 제어 영역을 추가한다.
- Prompt Studio 계열 밖에서도 쓸 수 있는 전용 wildcard 노드를 제공한다.

## 설계 원칙

- 기본 wildcard 문법과 랜덤 확장 동작은 Impact Pack을 참고한다.
- 구현 전 `ComfyUI-Impact-Pack/modules/impact/wildcards.py`,
  `ImpactWildcardProcessor`, `ImpactWildcardEncode`, `/impact/wildcards` API를
  확인한다.
- 가능한 경우 Impact Pack의 공개 함수나 안정적인 동작 계약을 사용하고,
  내부 구현을 그대로 복사하지 않는다.
- wildcard 확장은 node execution 단계에서 수행한다.
- 사용자가 입력한 원본 wildcard 텍스트는 live workflow에서 유지한다.
- 결과 이미지에 저장되는 workflow에는 확장된 텍스트를 기록해 같은 결과를 재현하게 한다.
- ComfyUI cache가 같은 wildcard 입력을 고정 결과로 재사용하지 않도록 seed/cache 정책을 명확히 한다.
- wildcard 파일이 없거나 비어 있으면 원문을 보존하고 실패하지 않는다.
- wildcard key path traversal은 차단하되, 사용자가 설정한 추가 root 경로는 명시적으로 허용한다.
- 기존 prompt ordering, trigger literal 보존, metadata filter 정책을 깨지 않는다.
- random mode와 sequential mode는 같은 wildcard 문법을 사용하되 선택 방식만 다르게 한다.
- sequential mode는 별도 sequence index가 아니라 현재 seed 값을 index driver로 사용한다.
- sequential mode는 seed after generate control을 자동 `increment`로 고정한다.
- 저장 이미지 workflow에는 seed 상태가 아니라 확장된 텍스트를 기록한다.

## 제안 문법

Impact Pack wildcard 기준을 기본값으로 둔다.

기준 소스:

- `D:\ComfyUI\ComfyUI_main\instances\ComfyUI_v0.25.1\custom_nodes\comfyui-impact-pack\modules\impact\wildcards.py`
- `D:\ComfyUI\ComfyUI_main\instances\ComfyUI_v0.25.1\custom_nodes\comfyui-impact-pack\modules\impact\impact_pack.py`

0.1.9 구현 전에는 위 로컬 v0.25.1 Impact Pack 코드를 다시 확인한다.
외부 문서보다 로컬 인스턴스 코드를 우선한다.

- file wildcard: `__wildcard-name__`
- dynamic prompt: `{a|b|c}`
- weighted dynamic prompt: `{2::a|5::b|3::c|d}`
- weighted file wildcard entry: `2::candidate`
- multi-select dynamic prompt: `{2$$a|b|c}`, `{1-3$$a|b|c}`, `{1-3$$, $$a|b|c}`
- multi-select wildcard source: `{1-3$$__colors__}`, `{1-3$$, $$__colors__}`
- wildcard quantifier: `3#__colors__`
- wildcard pattern: `__style/*__`, `__*/hair__`
- `.txt`, `.yaml`, `.yml` wildcard 파일
- seed 기반 deterministic 확장
- nested/transitive wildcard는 Impact Pack `process()`처럼 최대 반복 깊이를 제한해 처리한다.

EasyUse Anima 차이점:

- 기본 파일 위치는 Impact Pack의 `wildcards/` 또는 `custom_wildcards/`가 아니라
  ComfyUI 사용자 폴더의 `user\__easyuse_anima\wildcards`로 둔다.
- sequential wildcard는 EasyUse Anima 추가 기능으로 둔다.
- 저장 이미지 workflow 재현 정책은 Prompt Studio metadata 갱신 경로에 맞춘다.

### Inline wildcard

```text
{red|blue|green} hair
{2::red|5::blue|3::green|white} hair
```

- `|`로 후보를 구분한다.
- `N::candidate`는 weighted option이다.
- weighted option의 선택 확률은 `weight / sum(weights)`로 계산한다.
- weight가 없는 후보의 기본 weight는 `1`이다.
- 선택 후 결과에서는 앞의 `N::` prefix를 제거한다.
- dynamic prompt의 빈 후보는 Impact Pack처럼 빈 문자열 후보로 처리될 수 있다.
- 모든 weight 합이 유효하지 않은 경우의 fallback은 Impact Pack 동작을 따른다.
- 중첩 wildcard와 transitive wildcard는 Impact Pack처럼 반복적으로 펼친다.
- literal `{`, `}`가 필요하면 backslash escape를 사용한다.
- `{...}` 내부의 escaped brace는 dynamic prompt 경계로 보지 않는다.

### Multi-select dynamic prompt

```text
{2$$red|blue|green}
{1-3$$red|blue|green}
{1-3$$, $$red|blue|green}
{1-3$$__hair_color__}
```

- `N$$...`는 후보 중 N개를 선택한다.
- `A-B$$...`는 A개 이상 B개 이하를 선택한다.
- `-B$$...`는 1개 이상 B개 이하를 선택한다.
- `A-B$$separator$$...`는 선택 결과를 `separator`로 join한다.
- `...`가 단일 wildcard token이면 해당 wildcard 후보 목록을 multi-select 후보로 사용한다.
- multi-select에서도 `N::candidate` weight를 적용한다.
- 선택 결과에서는 weight prefix를 제거한다.

### File wildcard

```text
__hair_color__ hair
__style/anime__
__*/hair_color__
__style/*__
3#__hair_color__
```

- `__name__`은 wildcard 파일에서 한 줄을 선택한다.
- 하위 폴더는 `/` 또는 `\`로 표현할 수 있다.
- key normalize는 Impact Pack처럼 `\`를 `/`로 바꾸고, 공백을 `-`로 바꾸며, 소문자로 맞춘다.
- `..`, 절대 경로, drive letter는 허용하지 않는다.
- 확장자는 생략 가능하게 한다.
- 후보 파일명:
  - `name.txt`
  - `name.yaml`
  - `name.yml`
- `#`로 시작하는 줄과 빈 줄은 무시한다.
- wildcard 파일의 각 후보도 `N::candidate` weight를 가질 수 있다.
- `*`가 포함된 wildcard key는 matching wildcard 후보들을 합쳐서 선택한다.
- `__*/name__`은 `name`, `folder/name`, `name/sub`, `folder/name/sub`처럼 깊이에 상관없이 매칭한다.
- `/`가 없는 `__name__`이 직접 매칭되지 않으면 Impact Pack처럼 `__*/name__` fallback을 시도한다.
- `N#__name__` quantifier는 Impact Pack `RE_WildCardQuantifier` 동작을 기준으로 처리한다.

### YAML wildcard

```yaml
hair:
  - black hair
  - brown hair
colors:
  warm:
    - red
    - orange
  cold:
    - blue
    - green
```

- YAML의 list, dict, string, number 값을 wildcard 후보로 읽는다.
- nested dict는 slash key로 펼친다.
- 예: `colors/warm` -> `__colors/warm__`
- parent key는 하위 값을 합친 후보 목록으로 쓸 수 있게 한다.
- YAML 파일은 wildcard key가 파일 내부에 있으므로 자동완성과 cache 갱신에서 별도 파싱이 필요하다.

### Comment and recursion

- `#`로 시작하는 줄은 Impact Pack처럼 comment로 처리한다.
- dynamic prompt pass와 file wildcard pass를 반복해 transitive wildcard를 확장한다.
- 반복 깊이는 Impact Pack `process()`의 `replace_depth = 100` 정책을 기준으로 제한한다.

### LoRA syntax boundary

- `ImpactWildcardEncode`는 wildcard 확장 뒤 `<lora:name:weight>` 형식을 해석해 모델과 CLIP에 LoRA를 적용한다.
- 0.1.9의 Prompt Studio 계열과 전용 `Anima Wildcard` 노드는 기본적으로 string 확장 노드다.
- 따라서 `<lora:...>`는 텍스트로 보존하고, LoRA loader 동작은 재구현하지 않는다.
- 추후 MODEL/CLIP/CONDITIONING 출력 노드를 별도로 만들 때 `process_with_loras()` 호환을 검토한다.

## Wildcard Root와 추가 경로

wildcard 파일은 ComfyUI 사용자 폴더 아래 `user\__easyuse_anima`에서 관리한다.
Impact Pack은 `ComfyUI-Impact-Pack/wildcards`와 `custom_wildcards`를 사용하지만,
EasyUse Anima는 사용자 데이터가 노드팩 설치 경로에 묶이지 않도록 별도 user
영역을 기본 root로 쓴다.

```text
ComfyUI/
  user/
    __easyuse_anima/
      wildcards/
        hair_color.txt
        style/
          anime.txt
```

기본 root:

- `user\__easyuse_anima\wildcards`

추가 root:

- EasyUse Anima 설정에서 사용자가 wildcard 폴더 경로를 추가할 수 있게 한다.
- 설정 key 후보: `wildcard.extra_paths`
- UI label 후보: `Wildcard extra paths`
- 한국어 UI label 후보: `와일드카드 추가 경로`
- 입력 형식은 multiline text로 두고, 한 줄에 root 폴더 하나를 입력한다.
- absolute path와 ComfyUI root 기준 relative path를 모두 허용한다.
- 빈 줄과 `#`로 시작하는 줄은 무시한다.
- 존재하지 않는 경로는 경고만 남기고 무시한다.
- 기본 root는 항상 포함한다.

탐색 우선순위:

- 명시적으로 설정한 추가 root를 UI 순서대로 먼저 탐색한다.
- 기본 root는 마지막 fallback으로 탐색한다.
- 같은 wildcard key가 여러 root에 있으면 먼저 발견된 root의 값을 사용한다.
- `*` pattern wildcard는 모든 root의 matching key를 모아 후보로 사용하되, 같은 root/key 중복은 제거한다.
- 자동완성 후보도 같은 root 목록과 우선순위를 사용한다.
- 자동완성 응답은 wildcard 상대 key만 반환하고, 사용자 로컬 absolute path는 반환하지 않는다.

파일 wildcard `__hair_color__`는 아래 후보를 찾는다.

- `user\__easyuse_anima\wildcards\hair_color.txt`
- `user\__easyuse_anima\wildcards\hair_color.yaml`
- `user\__easyuse_anima\wildcards\hair_color.yml`

하위 폴더 wildcard `__style/anime__`는 아래 후보를 찾는다.

- `user\__easyuse_anima\wildcards\style\anime.txt`
- `user\__easyuse_anima\wildcards\style\anime.yaml`
- `user\__easyuse_anima\wildcards\style\anime.yml`

구현 시 확인할 사항:

- ComfyUI user folder를 안정적으로 찾는 방법
- `user\__easyuse_anima`가 없으면 자동 생성할지 여부
- EasyUse Anima 설정 UI에서 multiline path 설정을 저장하는 방법
- Windows 경로 구분자와 escape 처리
- wildcard 파일은 사용자 데이터로 취급하고 repo에 커밋하지 않는 정책

## Wildcard 설정 UI

Prompt Studio 계열과 전용 wildcard 노드의 기본 설정은 아래 항목으로 통일한다.

- `mode`
- `seed`
- `seed after generate` UI label: `시드 생성 후 제어`

Prompt Studio Advanced에서는 이 버튼과 제어 영역을 `mod guidance` 아래에 배치한다.

### Mode

- `일반 채우기`: queue 실행 시 wildcard를 확장한다. seed 정책에 따라 결과가 달라질 수 있다.
- `고정`: 같은 원문, 같은 seed, 같은 wildcard 파일 상태에서는 같은 확장 결과를 유지한다.
- `순차`: seed를 각 wildcard 후보 수로 나눈 나머지를 index로 사용한다. `시드 생성 후 제어`는 자동 `increment`로 운용한다.
- `재현`: Impact Pack처럼 한 번은 `고정`처럼 동작해 저장된 확장 결과를 재현하고, 이후 `일반 채우기`로 전환하는 방향으로 설계한다.

실제 내부 enum 이름은 구현 시 영어 identifier로 둔다.

후보:

- `populate`
- `fixed`
- `sequential`
- `reproduce`

### Seed와 선택 방식

- seed는 `Seed (rgthree)`처럼 사용자가 조작할 수 있게 한다.
- Prompt Studio 계열에 wildcard용 seed 입력과 seed control을 둔다.
- 후보 UI:
  - `wildcard_seed`
  - `wildcard_seed_control`
- control 후보:
  - `fixed`
  - `randomize`
  - `increment`
  - `decrement`
- 실제 widget 이름과 옵션명은 rgthree `Seed` 동작을 확인한 뒤 맞춘다.

### 일반 채우기 / 고정

- 기본 random wildcard 동작은 Impact Pack 동작을 기준으로 한다.
- 후보 선택은 seed 기반 deterministic random으로 둔다.
- dynamic prompt와 file wildcard 후보의 `N::candidate` weight를 Impact Pack과 같은 확률로 반영한다.
- `fixed` seed에서는 같은 입력이 같은 확장 결과를 만든다.
- `randomize`는 queue 실행마다 seed를 바꾼다.
- `increment` / `decrement`는 queue 실행 후 live workflow seed를 갱신한다.
- 저장 이미지 workflow에는 확장된 텍스트를 기록해 같은 결과를 재현하게 한다.

### Sequential wildcard

- sequential wildcard는 Impact Pack 기본 기능이 아니라 EasyUse Anima 확장 기능이다.
- sequential mode는 랜덤을 사용하지 않는다.
- 현재 seed 값을 index driver로 사용한다.
- 각 wildcard의 선택 index는 `seed % candidate_count`로 계산한다.
- queue 실행 후 live workflow seed를 1 증가시켜 다음 실행에서 다음 후보를 선택한다.
- 후보 목록 끝에 도달하면 modulo 계산으로 처음으로 순환한다.
- inline wildcard는 `{a|b|c}`의 후보 순서를 사용한다.
- file wildcard는 파일의 유효 줄 순서를 사용한다.
- weighted 후보도 후보 1개로 계산하고, 순차 모드에서는 weight를 확률로 사용하지 않는다.
- 예: `{2::a|5::b|3::c|d}`의 순차 후보 수는 4이고 출력 후보는 `a`, `b`, `c`, `d`다.
- 한 프롬프트에 wildcard가 여러 개 있으면 같은 seed 값을 각 후보 목록에 적용하고, 후보 수가 다르면 각 목록 길이로 modulo 처리한다.
- 후보 수가 0이면 원문 token을 보존한다.
- 순차 모드에서는 seed after generate control을 자동으로 `increment`로 바꾸고, UI에서도 다른 control을 선택할 수 없게 하는 방향을 우선 검토한다.
- 저장 이미지 workflow에는 seed가 아니라 확장된 텍스트를 기록한다.

결정 필요:

- 기본 mode를 `일반 채우기`로 둘지 `고정`으로 둘지
- `재현` 모드에서 wildcard 원문이 남아 있을 때 원문 보존과 고정 확장 중 어느 쪽을 우선할지
- 순차 모드에서 seed after generate control을 완전히 숨길지, 표시하되 잠글지

## 전용 Wildcard 노드

Prompt Studio 없이 wildcard만 확장할 수 있는 노드를 추가한다.

제안 노드:

- class id: `EasyUseAnimaWildcard`
- display name: `Anima Wildcard`
- category: `EasyUse Anima/Prompt`

입력 후보:

- `text`
- `mode`
- `seed`
- `seed_after_generate`

출력 후보:

- `text`: 확장된 prompt text
- `seed`: 실행에 사용된 seed 또는 다음 seed

동작:

- Prompt Studio 계열과 같은 wildcard parser/resolver를 사용한다.
- 저장 workflow 재현 정책도 Prompt Studio 계열과 같은 확장 결과 기록 방식을 따른다.
- 순차 모드는 `seed % candidate_count` 규칙을 그대로 사용한다.
- wildcard token이 없으면 입력 text를 그대로 출력한다.
- 전용 노드는 다른 노드에서 문자열 전처리기로 연결할 수 있어야 한다.

결정 필요:

- 출력 seed를 실행에 사용한 seed로 둘지, queue 후 갱신될 다음 seed로 둘지
- `STRING` 입력 연결과 multiline widget을 둘 다 지원할지 여부

## Wildcard 자동완성

Prompt Studio 계열 입력 UI와 전용 wildcard 노드의 text 입력에서 `__`를 입력하면 wildcard 후보를 표시한다.

- 자동완성 후보는 `wildcard.extra_paths`와 기본 `user\__easyuse_anima\wildcards` 목록을 합쳐 만든다.
- 후보 탐색 순서는 wildcard resolver와 동일하게 추가 root 우선, 기본 root fallback으로 맞춘다.
- 후보 표기는 확장자를 제거한 상대 경로로 한다.
- 삽입 형식은 `__relative/name__`로 한다.
- 하위 폴더는 `/` 표기를 기본으로 한다.
- `.txt`, `.yaml`, `.yml` 파일을 후보로 포함한다.
- 같은 basename의 여러 확장자가 있으면 우선순위를 정해야 한다.
- wildcard root가 없거나 추가 root가 유효하지 않으면 해당 root만 건너뛰고 실패하지 않는다.
- duplicate key는 우선순위가 높은 root의 후보만 표시한다.

API 후보:

- `GET /easyuse_anima/wildcards`
- 반환 예: `{"items": ["hair_color", "style/anime"]}`

프론트엔드 동작:

- 사용자가 `__`를 입력한 직후 또는 `__partial` 상태에서 후보를 연다.
- 선택하면 현재 token을 `__name__`으로 교체한다.
- 일반 tag 자동완성과 충돌하지 않게 wildcard token 내부에서만 동작한다.
- 목록 갱신은 파일 변경을 즉시 감시하지 않고, 처음에는 요청 시 refresh 또는 짧은 cache로 처리한다.

## 저장 workflow 재현 전략

### Prompt Builder / Prompt Studio

- 실행 시 원본 입력 텍스트에서 wildcard를 확장한다.
- 최종 출력과 metadata prompt는 확장된 텍스트 기준으로 만든다.
- result image workflow에는 각 prompt 입력 widget 값을 확장된 텍스트로 기록한다.
- live workflow는 원본 wildcard 입력과 다음 seed 상태를 유지한다.
- saved workflow는 확장 결과를 저장한다.

### Prompt Studio Advanced

- `mod guidance` 아래에 wildcard 제어 영역을 배치한다.
- 제어 항목은 `mode`, `seed`, `seed after generate`로 시작한다.
- 순차 모드 선택 시 `시드 생성 후 제어`를 자동 `increment`로 바꾼다.
- `advanced_fields`의 각 field `text`에 wildcard를 적용한다.
- 연결된 field socket 입력값에도 wildcard를 적용하되, 저장되는 `advanced_fields`에는 socket 입력 원문을 덮어쓰지 않는다.
- saved workflow에는 확장된 `advanced_fields`를 기록한다.
- NAIA fill과 wildcard가 같이 켜진 경우 순서는 `NAIA fill -> wildcard expansion -> prompt correction`으로 한다.
- NAIA 해상도 저장 정책은 기존 0.1.9 수정과 동일하게 유지한다.
- live workflow는 원본 `advanced_fields`와 다음 seed 상태를 유지한다.

### Prompt Studio Extend

- 활성 slot 값에 wildcard를 적용한다.
- hidden slot은 기존 정책대로 출력과 확장 대상에서 제외한다.
- saved workflow에는 확장된 slot 값을 기록한다.
- `fill_naia_prompt`와 wildcard가 같이 켜진 경우 순서는 `NAIA fill -> wildcard expansion -> prompt correction`으로 한다.
- live workflow는 원본 slot 값과 다음 seed 상태를 유지한다.

## 캐시 정책

- `wildcard_seed_control == randomize`이고 wildcard token이 존재하면 `IS_CHANGED`는 `float("nan")`을 반환한다.
- `wildcard_seed_control == increment/decrement`이면 queue 후 live seed 값을 갱신하고, stable change key에는 현재 seed를 포함한다.
- sequential mode이면 stable change key에는 현재 seed를 포함한다.
- fixed seed이면 seed, 원본 텍스트, wildcard 파일 목록 또는 mtime을 stable change key에 포함한다.
- reproduce mode이면 확장 완료 텍스트를 stable change key에 포함하고, wildcard 파일 변경은 cache key에 넣지 않는 방향을 우선 검토한다.
- wildcard token이 없으면 기존 cache 정책을 유지한다.
- 파일 wildcard는 기본 root와 추가 root의 파일 내용 변경을 cache key에 반영해야 한다.
- `wildcard.extra_paths` 설정값 변경도 wildcard list/cache key에 반영해야 한다.
