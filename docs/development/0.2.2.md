# 0.2.2 Idea Notes

This file is an early idea note for possible 0.2.2 work. It is not a release
announcement and does not mark the scope as final.

## Version Boundary

- `v0.2.1` is the current release tag before this note.
- Changes intended for 0.2.2 should stay in this file until the release scope
  is finalized.
- Do not move these ideas into `RELEASE.md` until implementation and validation
  are complete.

## Goals

- Simplify the Anima Prompt Corrector so it can be used like a normal
  multi-line string utility node.
- Explore a full all-in-one generation node that consumes Anima Prompt Data and
  runs the image generation path from prompt data to optional save.
- Keep beginner-facing controls compact while preserving advanced options for
  users who need detailed generation, patch, highres, and detailer settings.

## Track 1: Prompt Corrector Simple Node

Current state:

- `Anima Prompt Corrector` requires `prompt`, `artist_overrides`, and
  `artist_exclusions`.
- It returns both `corrected_prompt` and a JSON `report`.
- The extra inputs and report socket are useful for debugging, but make the
  node less convenient as a simple string-processing node.

Idea:

- Add a separate simplified node instead of changing the existing node contract.
- Candidate display name: `Anima Prompt Corrector Simple`.
- Required input:
  - `prompt`: multi-line `STRING`
- Output:
  - one `STRING` socket, for example `prompt` or `text`
- Behavior:
  - Reuse the same correction pipeline and current tag policy.
  - Do not output the JSON report.
  - Keep manual artist override and exclusion controls out of the main surface.
  - If override/exclusion support is still needed, keep it as hidden, optional,
    or a separate advanced node rather than adding visible required sockets.

Compatibility rules:

- Do not remove or reorder outputs from the existing `Anima Prompt Corrector`.
- Do not make old workflows migrate automatically.
- The simple node should be an additive convenience node.

Validation targets:

- Unit test that the simple node returns the same corrected prompt text as the
  existing node with empty overrides and exclusions.
- Unit test that it has exactly one output socket.
- Locale entries and docs should describe it as the default string-pipeline
  corrector, while keeping the existing node as the report/debug variant.

## Track 2: Full All-In-One Generation Node

Working idea:

- Add a new node that accepts `EASYUSE_ANIMA_PROMPT_DATA` plus the required
  generation inputs and produces the final image.
- This should be a new node or node family, not a replacement for existing
  Prompt Studio, Prompt Data Conditioning, SAM3, or workflow template pieces.

Candidate display names:

- `Anima Prompt Data Generator`
- `Anima AiO Generator`
- `Anima Image Generator`

Primary input model:

- Required visible inputs should stay small:
  - `model`
  - `clip`
  - `vae`
  - `EASYUSE_ANIMA_PROMPT_DATA`
  - `seed`
  - core sampler preset or sampler/scheduler pair
  - `steps`
  - `cfg`
  - base denoise or generation mode
- Resolution should default to Prompt Data width/height.
- Advanced generation inputs should be grouped or folded away:
  - Anima Mod Guidance mode and profile
  - artist mix mode and tuning
  - highres enable, scale, steps, denoise, upscaler
  - model patch and optimization options
  - SAM3 detailer enable, targets, threshold, detailer sampler settings
  - metadata and save options

Pipeline sketch:

1. Normalize `EASYUSE_ANIMA_PROMPT_DATA`.
2. Read prompt, negative prompt, width, height, artist mix, and Mod Guidance by
   dict key.
3. Apply model patch options.
4. Encode positive and negative conditioning.
5. Create or receive the base latent.
6. Run the base KSampler stage.
7. Optionally run highres upscale and second KSampler stage.
8. Optionally run SAM3 detailer passes.
9. Decode final image.
10. Optionally save the image with prompt data and workflow metadata.
11. Return image and any useful optional diagnostics.

Preview direction:

- UI design should be drafted before backend implementation.
- First draft asset: `docs/images/aio-generator-ui-draft.svg`.
- Rendered preview: `docs/images/aio-generator-ui-draft.png`.
- Preview should be selectable, for example:
  - off
  - prompt summary
  - generation stage summary
  - latest output image
  - comparison or detailer stage preview if feasible
- The preview UI should not trigger heavy work from typing, dragging, or layout
  changes.
- Saved workflow metadata should still reproduce the prompt data used for the
  generated image.

Implementation direction:

- Prefer orchestration over reimplementing ComfyUI core nodes.
- Use existing EasyUse Anima helpers for prompt data, artist mix, Mod Guidance,
  SAM3 context/detailer, and metadata handling.
- Use Impact Pack delegation for detailer loops and SEGS conversion.
- Keep model patch and detailer dependencies optional at runtime where
  possible, with clear errors when a selected feature is unavailable.
- Keep the node cache behavior deterministic unless a selected option requires
  fresh generation.

Beginner UX direction:

- Default mode should generate an image from Prompt Data with minimal visible
  controls.
- Main controls should answer common questions quickly:
  - what seed?
  - how many steps?
  - what sampler?
  - is highres on?
  - are face/eye detailers on?
  - should the result be saved?
- Advanced controls should be available but not required for the first working
  generation.

Compatibility and risk notes:

- Do not collapse the existing Prompt Data node family too early.
- Do not break workflows that already connect `Anima Prompt Data Conditioning`
  to normal sampler and decode nodes.
- Do not hide required generation state only in frontend DOM widgets; queued
  prompts and saved workflows must carry all required values.
- Store expandable AiO resource and generation parameters in versioned JSON
  fields, then migrate/default-merge them by key. Do not rely on adding many
  ordered widgets for settings that are expected to change across versions.
- Keep the dedicated `easy use anima input` socket name stable across locales so
  workflow links and user mental models do not drift.
- Do not reimplement Impact Pack `DetailerForEach` or `MaskToSEGS`.
- Treat image save metadata as a correctness requirement, not a UI convenience.

## Proposed Work Order

1. Draft the all-in-one node UI layout first.
2. Decide the exact node names and visible input set.
3. Implement the Prompt Corrector simple node as a small additive change.
4. Add tests and docs for the simple node.
5. Build a backend prototype of the all-in-one node without custom frontend UI.
6. Add UI foldouts and preview behavior after the backend contract is stable.
7. Add a small example workflow only after saved workflow reload and queue
   validation pass.

## Open Questions

- Should the all-in-one node accept separate `model`, `clip`, and `vae`, or a
  context object from an upstream loader?
- Should image input / img2img be part of the first version, or added after the
  text-to-image path is stable?
- Should save behavior call ComfyUI's normal save path or provide a project
  metadata wrapper around it?
- Should highres be latent-upscale first, image-upscale first, or preset-based?
- Which SAM3 detailer targets should be visible by default?
- Should the full node output only `IMAGE`, or also expose intermediate image,
  latent, prompt data, and metadata outputs?
